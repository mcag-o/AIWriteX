# AIWriteX 内容中站化重构 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 将当前以桌面软件为中心的 AIWriteX 改造为中心化、无状态、可服务化部署的“内容中站”系统，仅保留可复用的业务核心能力，剥离桌面端 UI、软件分发、授权壳、托盘与外观资源。

**Architecture:** 采用“配置中枢 + 中台服务层 + 工作流内核 + 可组合节点运行时 + 平台接入/发布适配层 + 模板与内容资产层”的分层架构。核心原则是先抽离纯业务逻辑，再解除对 FastAPI Web UI、pywebview、multiprocessing UI 任务管理、发布态路径分支和桌面专属日志模型的耦合，最终形成可 API 化、可任务化、可扩展的内容中站。

**Tech Stack:** Python 3.11/3.12, FastAPI, Pydantic, Jinja2/模板资产, CrewAI（阶段性保留，后续可替换）, requests/httpx, PyYAML/tomlkit, structured logging, optional Celery/RQ/Arq for async jobs.

---

## 一、目标边界与重构原则

### 1.1 转型后的目标形态

- 目标系统不再是桌面应用，而是一个可由 Web 管理台、脚本、第三方服务或调度器调用的“内容中站”。
- 中站负责统一提供以下能力：
  - 配置中枢
  - 内容生成层
  - 工作流编排内核
  - 可组合节点能力
  - 风格化/二次改写能力
  - 模板库与格式化能力
  - 平台接收与发布能力
  - 发布记录与作业状态管理
  - 通用工具、日志、接口契约
- 中站本体必须支持“无桌面环境部署”和“多入口调用”。

### 1.2 保留原则

- 保留“纯业务逻辑”与“可复用能力抽象”。
- 保留模板库与公众号发布等有业务价值的资产。
- 保留现有 FastAPI 中真正代表服务能力的 API 思路，但不保留为桌面壳服务的页面与状态管理方式。
- 保留现有工作流配置抽象，但弱化对 CrewAI 的直接绑定。

### 1.3 丢弃原则

- 丢弃所有桌面壳、窗口、托盘、图标、UI 设置、软件分发、发布态资源复制逻辑。
- 丢弃所有仅服务于本地 GUI 交互的模板、静态文件、WebSocket 日志推送、窗口状态、主题模式、更新检查、帮助页、外链打开等逻辑。
- 丢弃含附加授权限制且无法安全复用的代码，优先重写，不做直接迁移。

## 二、现状模块梳理与价值判断

### 2.1 高价值保留模块

#### 配置中枢
- `src/ai_write_x/config/config.py`
- `src/ai_write_x/config/config.yaml`
- `src/ai_write_x/config/aiforge.toml`

价值：
- 已形成统一配置入口、默认值、校验、保存逻辑。
- 已覆盖模型、模板、平台、生成参数、公众号凭据、创意参数等。

问题：
- 配置类过于庞大，混合了 UI 配置、发布态路径、模板、平台、创意、运行参数。
- 采用单例 + 文件直接读写，不利于服务化与测试。

结论：
- 保留“配置域模型”与“配置装载”思路。
- 重构为分域配置，不保留当前单体 `Config` 直接作为中枢实现。

#### 工作流内核
- `src/ai_write_x/core/base_framework.py`
- `src/ai_write_x/core/content_generation.py`
- `src/ai_write_x/core/unified_workflow.py`
- `src/ai_write_x/core/tool_registry.py`
- `src/ai_write_x/core/system_init.py`

价值：
- 已形成生成、创意、格式转换、保存、发布的完整链路。
- 已提供 WorkflowConfig、AgentConfig、TaskConfig、ContentResult 等核心抽象。

问题：
- `UnifiedContentWorkflow` 直接耦合平台适配器、Config 单例、保存逻辑、发布逻辑、创意逻辑。
- `ContentGenerationEngine` 与 CrewAI 强绑定。
- `system_init.py` 是硬编码装配，不具备中站插件式装配能力。

结论：
- 作为重构核心保留。
- 需要拆成 orchestrator / runtime / ports / services 四层。

#### 创意改写层
- `src/ai_write_x/creative/dimensional_engine.py`

价值：
- 具备成熟的维度化改写与风格化能力，是中站差异化资产。

问题：
- 内部直接 new `ContentGenerationEngine`，与运行时绑定过深。

结论：
- 保留业务思想与配置结构。
- 抽成“风格化节点”或“rewrite service”。

#### 平台适配层
- `src/ai_write_x/adapters/platform_adapters.py`

价值：
- 平台类型、格式化、发布抽象清晰。

问题：
- `PlatformAdapter` 依赖 `web.api.articles.save_publish_record`，反向耦合 UI/API 层。
- 多平台实现质量不均，只有微信相对完整。

结论：
- 保留接口思想与微信实现骨架。
- 反转依赖，平台层不得依赖 API 层。

#### 工具能力层
- `src/ai_write_x/tools/custom_tool.py`
- `src/ai_write_x/tools/wx_publisher.py`

价值：
- 模板读取、搜索工具、微信发布器属于高复用业务能力。

问题：
- `custom_tool.py` 同时承担 CrewAI tool 包装、文件读取、搜索格式化。
- `wx_publisher.py` 过大，混合 token、封面图、素材上传、草稿、发布轮询。

结论：
- 拆分并保留。

#### 工具与基础设施
- `src/ai_write_x/utils/log.py`
- `src/ai_write_x/utils/path_manager.py`
- `src/ai_write_x/utils/utils.py`
- `src/ai_write_x/utils/content_parser.py`
- `src/ai_write_x/utils/comm.py`

价值：
- 有现成的日志、路径、内容解析、通用辅助能力。

问题：
- 明显服务桌面模式与多进程 UI 推送。
- 路径管理依赖“release vs dev”双分支，不适合服务化。

结论：
- 部分逻辑保留并重写。
- `comm.py` 大概率可废弃。

#### 模板资产
- `knowledge/templates/**`

价值：
- 是业务资产，不是代码资产。

问题：
- 目前模板访问与 UI 分类管理耦合。

结论：
- 完整保留，重构为模板资产库。

### 2.2 中价值、需谨慎处理模块

#### 平台接收/文章管理
- `src/ai_write_x/web/api/articles.py`

价值：
- 包含文章列表、读取、预览、发布记录、平台列表等服务化可用逻辑。

问题：
- 路由层与文件系统、发布逻辑、记录逻辑耦合在一起。

结论：
- 保留为“内容资产服务 + 发布记录仓储”的来源，不直接保留文件。

#### 生成 API
- `src/ai_write_x/web/api/generate.py`

价值：
- 提供生成任务入口和任务状态接口雏形。

问题：
- 完全围绕桌面前端任务控制设计，依赖多进程、全局变量、WebSocket 日志。

结论：
- 仅保留接口语义与请求结构参考。
- 运行机制必须重写为中站作业系统。

### 2.3 低价值或应废弃模块

## 三、废弃清单与详细分析

### 3.1 必须废弃的代码与资源

#### A. 桌面入口与授权壳
- 丢弃：`main.py`
- 丢弃：`src/ai_write_x/license/__init__.py`

原因：
- 两者职责完全是桌面软件启动壳和商业授权插口。
- 对中站服务化没有业务价值。

替代：
- 新入口改为 `apps/api/main.py` 或 `apps/worker/main.py`。

#### B. pywebview 桌面容器与系统托盘
- 丢弃：`src/ai_write_x/web/webview_gui.py`
- 丢弃：`src/ai_write_x/utils/tray_manager.py`
- 丢弃：`src/ai_write_x/utils/icon_manager.py`

原因：
- 纯桌面交互层。
- Linux/Windows 打包兼容性逻辑会污染服务化架构。

#### C. 桌面 UI 资源
- 丢弃：`src/ai_write_x/web/static/**`
- 丢弃：`src/ai_write_x/web/templates/**`
- 丢弃：`src/ai_write_x/assets/UI/**`

原因：
- 这些是桌面软件外观资源，不属于内容中站核心资产。
- 用户已明确要求全部丢弃。

#### D. UI 状态与前端辅助 API
- 丢弃：`src/ai_write_x/web/state.py`
- 丢弃：`src/ai_write_x/web/api/config.py` 中以下能力：
  - `get_ui_config`
  - `save_ui_config`
  - `get_help_manual`
  - `check_for_updates`
  - `open_external_url`

原因：
- 全是前端视图态、软件设置或分发升级辅助能力。

#### E. 发布态资源路径分支
- 清理：`src/ai_write_x/utils/path_manager.py` 中与 release 包复制模板、系统目录适配相关逻辑

原因：
- 中站部署应采用统一环境变量和挂载卷策略，而不是桌面用户目录策略。

### 3.2 建议重写而非直接迁移的模块

#### A. `src/ai_write_x/config/config.py`
原因：
- 文件过大，领域边界不清。
- 带单例和直接文件副作用。

策略：
- 按 domain 拆分为 `llm`, `workflow`, `publish`, `template`, `storage`, `rewrite` 六类配置模型。

#### B. `src/ai_write_x/web/api/generate.py`
原因：
- 多进程、WebSocket、全局状态设计不适合中站。

策略：
- 重写为 job-oriented API：`POST /jobs`, `GET /jobs/{id}`, `POST /jobs/{id}/cancel`, `GET /jobs/{id}/events`。

#### C. `src/ai_write_x/web/api/articles.py`
原因：
- 路由层混入文件读写、发布逻辑和记录逻辑。

策略：
- 拆成 `content repository`, `publish record repository`, `content service`, `publish service`。

#### D. `src/ai_write_x/utils/log.py`
原因：
- 以 UI 队列转发为中心，不适用于服务端统一日志。

策略：
- 保留部分通用能力，重写为结构化日志系统。

#### E. `src/ai_write_x/tools/custom_tool.py`
原因：
- 工具包装与具体实现混杂。

策略：
- 拆为 `template reader`, `search provider`, `crewai adapters` 三层。

### 3.3 暂不迁移、待授权确认的模块

- `src/ai_write_x/tools/hotnews.py`
- `src/ai_write_x/tools/search_template.py`

原因：
- 文件头明确声明附加授权限制。
- 对中站有参考价值，但不建议直接原样迁移到新可服务化产品。

策略：
- 第一阶段不迁移代码，仅迁移接口契约与业务需求。
- 后续用新实现替代：
  - 热点采集：改为 provider interface + 新抓取实现
  - 参考文章抽取：改为通用 extractor pipeline

## 四、目标架构方案

### 4.1 总体分层

建议采用 6 层架构：

1. **interfaces**：对外 API、管理后台接口、平台接收入口、CLI
2. **application**：作业编排、服务用例、命令对象、DTO、权限与事务边界
3. **domain**：内容实体、模板实体、工作流定义、节点协议、发布结果、配置模型
4. **runtime**：工作流执行器、节点注册中心、LLM 驱动适配、任务运行时
5. **infrastructure**：文件仓储、日志、HTTP 客户端、微信 API、模板存储、配置存储
6. **assets**：模板库、默认配置、schema、提示词模板

### 4.3 模块依赖关系

依赖方向必须单向：

```text
interfaces -> application -> domain
application -> runtime
runtime -> domain
runtime -> infrastructure
application -> infrastructure
infrastructure -> domain
assets: 被 infrastructure/runtime 读取，但不反向依赖代码层
```

关键约束：
- `domain` 不得依赖 FastAPI、CrewAI、requests、文件系统。
- `runtime` 不得依赖桌面 UI 或 `web.api.*`。
- `platform adapters` 不得依赖 API 层。
- `logging`、`storage`、`publish record` 必须下沉到 infrastructure。

## 五、功能分组抽取方案

### 5.1 配置中枢

目标：将当前单体 `Config` 拆为可组合配置域。

建议拆分：
- `HubSettings`：全局运行配置
- `LLMSettings`：模型供应商、模型名、token、api_base
- `WorkflowSettings`：生成字数、格式、默认平台、自动发布、作业策略
- `RewriteSettings`：二次改写、风格化、维度选择
- `TemplateSettings`：模板目录、压缩策略、默认分类
- `PublishSettings`：公众号凭据、平台凭据、封面图策略
- `StorageSettings`：内容目录、模板目录、日志目录、临时目录

技术建议：
- 使用 Pydantic Settings 或 dataclass + schema validation。
- 支持三层配置源：默认值 < 文件 < 环境变量/密钥注入。
- 不再采用全局单例作为唯一访问方式，改为显式依赖注入。

### 5.2 中台化组件

目标：形成可被 API、CLI、调度器复用的用例服务。

建议服务：
- `ContentService`：内容读取、保存、查询、预览转换
- `WorkflowService`：创建与执行工作流
- `TemplateService`：模板分类、模板读取、模板元数据
- `PublishService`：平台发布与记录查询
- `IngestionService`：参考内容抽取、热点接收、素材导入
- `ConfigService`：配置读取与校验

### 5.3 生成层核心

来源：
- `src/ai_write_x/core/content_generation.py`
- `src/ai_write_x/core/agent_factory.py`
- `src/ai_write_x/utils/content_parser.py`

改造要点：
- 把“工作流定义”和“执行引擎”分开。
- `ContentGenerationEngine` 变为 `GenerationEngine`，仅依赖抽象 `LLMDriver`。
- `AgentFactory` 重构为 `ExecutionUnitFactory`，允许 driver=crewai/native。
- `ContentResult` 升级为统一内容对象，包含：body, title, summary, metadata, provenance。

### 5.4 工作流内核

来源：
- `src/ai_write_x/core/unified_workflow.py`
- `src/ai_write_x/core/base_framework.py`

改造方向：
- 将当前串行函数式工作流拆为节点图：
  - `InputNode`
  - `SearchNode`
  - `GenerateNode`
  - `RewriteNode`
  - `FormatNode`
  - `PersistNode`
  - `PublishNode`
- 用 `WorkflowDefinition + NodeSpec + ExecutionContext` 描述流程。
- 支持可配置编排而非写死在 Python 逻辑里。

### 5.5 外部能力封装为可组合节点

来源：
- `src/ai_write_x/tools/custom_tool.py`
- `src/ai_write_x/tools/search_template.py`
- `src/ai_write_x/tools/hotnews.py`

目标：
- 外部能力不再作为“CrewAI 工具类”存在，而是作为可组合节点/服务存在。

节点分类：
- `HotTopicNode`
- `ReferenceExtractNode`
- `SearchNode`
- `TemplateReadNode`
- `TemplateFillNode`
- `WeChatPublishNode`

建议：
- 节点协议统一为 `execute(context) -> context_patch`。
- 支持 registry 动态注册。
- 若未来替换 CrewAI，节点层无需变化。

### 5.6 公众号发布管理逻辑

来源：
- `src/ai_write_x/tools/wx_publisher.py`
- `src/ai_write_x/adapters/platform_adapters.py`
- `src/ai_write_x/web/api/articles.py`

改造拆分：
- `wechat_client.py`：微信 API 原子调用
- `wechat_media_service.py`：图片/素材上传
- `wechat_draft_service.py`：草稿管理
- `wechat_publish_service.py`：发布与轮询
- `publish_record_repository.py`：发布记录持久化
- `wechat_adapter.py`：平台层统一入口

关键目标：
- 发布逻辑脱离文章 API 路由。
- 发布记录写入 repository，而非从 adapter 反向调用 API 层。
- 支持单账号、多账号、批量文章、多作业发布。

### 5.7 二次改写 / 风格化处理层

来源：
- `src/ai_write_x/creative/dimensional_engine.py`

改造目标：
- 抽成独立 `RewriteEngine`。
- 支持以下模式：
  - preserve rewrite
  - style transfer
  - anti-AI flavor（若后续自研）
  - audience adaptation
- 输入输出统一为 `ContentDocument`。

### 5.8 平台内容接收

定义：中站需要接收外部输入内容，而不是只从 UI 表单接收。

建议接收入口：
- `POST /ingestion/reference-urls`
- `POST /ingestion/raw-content`
- `POST /ingestion/hot-topics`
- `POST /workflows/execute`

说明：
- 当前项目没有成熟“平台回调接收”能力，需新建。
- 第一阶段只做主动接收，不做 webhook 复杂回调。

### 5.9 杂项工具函数集合

来源：
- `src/ai_write_x/utils/utils.py`
- `src/ai_write_x/utils/content_parser.py`
- `src/ai_write_x/utils/path_manager.py`

## 六、数据与逻辑迁移策略

### 6.1 总体迁移原则

- 采用“先复制抽取，再重定向切换，最后删除旧壳”的三段式迁移。
- 不直接在旧模块上持续打补丁。
- 所有核心业务迁移时先做接口隔离层，避免一次性重写失控。

### 6.2 平滑抽取策略

#### 阶段 A：建立新骨架，不删旧系统
- 新建 `content_hub` 包。
- 先复制抽象类型与高价值纯逻辑到新目录。
- 在旧系统上仅做最少桥接，不做大改。

#### 阶段 B：迁移核心链路
- 先迁移：配置模型、工作流实体、内容生成引擎、模板服务、微信发布器。
- 旧的 `crew_main.py` 和 `web/api/*.py` 变为“适配旧接口调用新服务”。

#### 阶段 C：切断旧依赖
- 当新 API 可独立运行后，删除 `main.py`、`license`、`webview_gui.py`、静态资源与 UI 模板。

### 6.3 无状态化策略

当前有状态耦合点：
- `Config.get_instance()` 单例
- `web/state.py` 全局状态
- `generate.py` 中全局 `_current_process`, `_task_status`
- 日志队列与进程内状态

目标改法：
- 配置通过请求上下文或容器注入，不以单例共享状态。
- 作业状态写入 `JobRepository`，而非内存全局变量。
- 工作流执行函数设计为纯输入/输出：
  - input = request + settings + repositories + adapters
  - output = content result + events + publish result
- 日志作为事件写入，不依赖 websocket queue。

### 6.4 存储迁移策略

建议抽象三类仓储：
- `ContentRepository`
- `TemplateRepository`
- `PublishRecordRepository`

第一阶段实现：
- 基于文件系统保留现状兼容性。

第二阶段演进：
- 发布记录和作业状态迁移至 SQLite/PostgreSQL。

### 6.5 API 平滑兼容策略

建议保留并兼容以下语义，但更换实现：
- 配置获取/更新
- 模板分类/列表
- 文章列表/内容读取
- 触发生成
- 发布文章

不兼容删除：
- 根页面 `/`
- UI config
- websocket UI logs
- `/shutdown`
- 更新检查与外链打开

## 八、风险评估

### 8.1 高风险

#### 风险 1：核心逻辑深度依赖 Config 单例
- 影响：迁移时大量隐式依赖失效
- 缓解：先做兼容 facade，再逐步改显式注入

#### 风险 2：CrewAI 绑定导致节点化重构困难
- 影响：工作流边界无法清晰拆出
- 缓解：先保留 CrewAI 作为 `LLMDriver` 的一种实现，不立即去除

#### 风险 3：微信发布逻辑与文章记录分散
- 影响：容易出现记录丢失、状态不一致
- 缓解：先定义统一发布服务与记录仓储，再接入 API

#### 风险 4：附加授权文件不可直接迁移
- 影响：热点采集和参考文章抽取能力可能短期缺失
- 缓解：第一阶段用占位接口替代，第二阶段重写实现

### 8.2 中风险

#### 风险 5：模板资产与平台适配强耦合
- 影响：模板渲染流程迁移后效果可能变化
- 缓解：保留模板读取/压缩/渲染链路的黄金路径回归测试

#### 风险 6：旧 API 与新中台接口同时存在带来维护成本
- 缓解：设置过渡期，旧 API 只做转发，不新增功能

### 8.3 低风险

#### 风险 7：日志系统重写导致历史 UI 日志不可兼容
- 缓解：中站不需要兼容 UI websocket 语义，可直接升级

## 九、技术建议

### 9.1 服务化建议

- 保留 FastAPI 作为管理与调用 API 层。
- 生成任务使用异步作业模型，不采用当前的直接多进程 + 全局变量方案。
- 若规模不大，第一阶段可使用 `asyncio + background tasks + file/db job store`。
- 若后续并发与任务隔离要求更高，升级为 `Celery/RQ/Arq`。

### 9.2 模块化建议

- 每个能力域必须有：domain model、service、repository/adapter interface。
- 运行时与业务规则隔离，任何工作流节点都不应直接读 `Config.get_instance()`。
- 平台发布器必须通过 `PublishAdapter` 接口统一调用。

### 9.3 扩展性建议

- 工作流定义尽量数据化，支持 YAML/JSON 配置节点链。
- 节点注册表支持插件化，便于未来接入 Notion、飞书、XHS、知识库、RAG。
- 模板库引入元数据索引，支持标签、平台、主题、风格筛选。

### 9.4 可测试性建议

- 对 `domain` 和 `application` 做纯单元测试。
- 对微信发布和参考抽取做 contract/integration tests。
- 对关键工作流做 golden output tests。
- 为模板渲染结果保留 snapshot tests。

### 9.5 可观测性建议

- 每个 workflow/job 生成唯一 `trace_id` / `job_id`。
- 记录节点耗时、失败点、输入输出摘要。
- 发布动作记录账号、文章、平台、状态、错误码。
